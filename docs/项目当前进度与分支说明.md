# 项目当前进度与分支说明

## 1. 当前总体进度

本项目基于 RuoYi-Vue 3.2.0，目标是建设“权限治理 + RAG 文件入库 + 安全检索 + 审计追踪”的权限控制 RAG 平台。

当前已完成：

- 若依平台基座迁移与本地运行环境适配；
- 用户组、用户组关系、权限策略、策略绑定等权限治理基础模块；
- 安全中心能力，包括访问监控、静态规则、IP 黑名单与拦截；
- RAG 文件入库真实链路；
- 平台侧文档权限标签回写；
- RAG 入库审计；
- RAG 安全检索 mock 闭环；
- 平台侧远程 RAG 检索适配分支。

当前尚未完成：

- RAG Server 真实 /rag/search 检索接口；
- 端到端真实安全 RAG 检索联调。

## 2. 已完成链路

### 2.1 RAG 文件入库真实链路

当前已跑通：

若依前端 RAG 文件入库页面
→ 若依后端 RagFileProxyController
→ RAG Server 8081
→ MariaDB sys_rag_file
→ MinIO 原文件备份
→ Milvus 文档切块向量
→ 若依平台侧 sys_rag_doc 权限标签回写
→ RAG 入库审计日志

已验证 username.txt 可成功写入 MariaDB、MinIO 和 Milvus。

### 2.2 平台侧安全检索 mock 闭环

main 分支已完成：

当前登录用户
→ PermissionContextService.buildContext()
→ PolicyDecisionService.decide()
→ 生成 metadataFilter
→ 从 sys_rag_doc 模拟候选结果
→ RagSecondFilterService 二次权限过滤
→ RagAuditLog 检索审计
→ 返回 rawResults / filteredResults / rejectedResults

当前二次过滤规则：

- PUBLIC 文档默认允许登录用户访问；
- 用户具备对应 scopeCode 时允许访问；
- 用户不具备对应 scopeCode 时拒绝返回；
- 被拒绝结果进入 rejectedResults，用于前端展示和审计解释。

包含：

### 3.2 fufu_week3

当前包含：

- 文件上传接口；
- MariaDB 文件元数据写入；
- MinIO 原文件备份；
- LangChain4j 文档切分；
- Milvus 文档切块写入；
- MariaDB / Milvus / MinIO 三端列表查询接口。

当前已完成入库侧，尚未完成：

- POST /rag/search

也就是还没有真实检索接口。

### 3.3 feature/platform-remote-search

定位：平台侧真实检索联调准备分支。

当前新增：

- RagSearchRequest 支持 topK；
- RagSearchRequest 支持 useRemote；
- 新增 IRagRemoteSearchService；
- 新增 RagRemoteSearchServiceImpl；
- RagSearchController 支持 mock / remote 双模式。

请求示例一：

{
  "query": "研发制度",
  "topK": 5,
  "useRemote": false
}

表示继续走 sys_rag_doc mock 检索。

请求示例二：

{
  "query": "研发制度",
  "topK": 5,
  "useRemote": true
}

表示调用 RAG Server：

POST {rag.server.url}/rag/search

并携带 query、topK、userContext、metadataFilter 和 platformFilterMode。

## 4. fufu 下一步任务

fufu 需要在 fufu_week3 分支补真实检索接口：

POST /rag/search
Content-Type: application/json

请求体建议：

{
  "query": "查询内部研发制度",
  "topK": 5,
  "userContext": {
    "userId": 1,
    "userName": "admin",
    "admin": true,
    "groupCodes": ["default"],
    "scopeCodes": ["INTERNAL"]
  },
  "metadataFilter": "scope_code in [\"INTERNAL\"]",
  "platformFilterMode": "metadata_filter_and_second_filter"
}

返回体建议：

{
  "code": 200,
  "msg": "操作成功",
  "data": {
    "query": "查询内部研发制度",
    "costTime": 123,
    "rawResultCount": 5,
    "results": [
      {
        "docId": "file_id",
        "title": "文件名",
        "content": "命中的文本切块内容",
        "scopeCode": "INTERNAL",
        "level": "INTERNAL",
        "score": 0.92,
        "chunkId": "chunk_id",
        "fileId": "file_id"
      }
    ]
  }
}

最低要求：

1. 接收 query 和 topK；
2. 支持 scopeCodes 或 metadataFilter 中的 scope_code 过滤；
3. 从 Milvus 返回候选 chunk；
4. 每条结果必须带 docId、title、content、scopeCode、level、score、chunkId、fileId。

如果暂时无法完成 query embedding + Milvus search，可以先做过渡版：

接收 query / topK / scopeCodes
→ 从 Milvus 已有 chunk 中查询
→ 按 scope_code 过滤
→ 返回符合格式的 results

后续再补真正的向量相似度检索。

## 5. 后续联调流程

当 fufu 完成 /rag/search 后，平台侧按以下流程联调：

1. 启动 MariaDB、Redis、MinIO、Milvus；
2. 启动 fufu 的 RAG Server；
3. 启动若依后端；
4. 启动若依前端；
5. 先上传测试文件，确认入库成功；
6. 使用 useRemote=false 验证 mock 检索；
7. 使用 useRemote=true 验证真实 RAG Server 检索；
8. 检查 rawResultCount、filteredResultCount、rejectedResultCount、filteredResults、rejectedResults 和 RagAuditLog。
